📖 Overview

This application lets you create Zendesk end-user accounts and authenticate them into the Zendesk messaging widget using JWT (JSON Web Token) authentication. It acts as a backend proxy — all Zendesk and Sunshine Conversations API calls are made server-side, keeping your credentials and JWT signing secret out of the browser.

🔑 Credential Requirements

You need two sets of credentials from your Zendesk account:

Zendesk Admin API Credentials

Used to create end-user accounts and set passwords via the Zendesk REST API (/api/v2/).

FieldDescription
SubdomainYour Zendesk account subdomain (e.g., z3nbwilliams from z3nbwilliams.zendesk.com)
Admin EmailThe email address of a Zendesk admin account
API TokenA Zendesk API token generated in Admin Center → Apps → Zendesk API → API Tokens

👉 Zendesk: Managing API tokens in the Admin Center

Sunshine Conversations (Sunco) Credentials

Used to create/update Sunco user profiles and sign JWT tokens for messenger authentication. Generated from Conversations API keys.

FieldDescription
App IDThe App ID from the Conversations API key token
Key IDThe Key ID from the Conversations API key token (format: app_…)
Secret KeyThe Secret Key from the same Conversations API key token (used as a raw string for HMAC-SHA256 signing — do NOT base64-decode it)
Widget KeyFrom Admin Center → Channels → Messaging → Web Widget → Installation

👉 Zendesk: Using the Conversations API keys

⚠️ Secret Key note: The JWT Secret Key is used exactly as shown in the Admin Center — as a raw UTF-8 string for HMAC-SHA256 signing. Do not base64-decode it before signing. The JWT scope must be "user" (not "end_user").

🛡️ Configuration Page

The Configuration page has three steps after saving credentials:

1
Create Zendesk End-User — Enter a full name and email address. The external ID is automatically generated from the email. The app calls POST /api/v2/users with role: 'end-user', verified: true, and skip_verify_email: true so the user can immediately authenticate without email verification. After creation, the fields are locked to prevent accidental changes.
2
Set Password — Sets a password for the newly created user via POST /api/v2/users/:id/password. The password is required to authenticate the user.
3
Login & Open Widget — Creates a Sunco user profile, generates a JWT, and authenticates into the messaging widget. The widget opens automatically using the Widget Key configured in step 1.
💡 Credentials entered on this page are stored in the server's in-memory session store (keyed by a random sessionId). They are never saved in the browser. The sessionId is the only thing stored in sessionStorage. Sessions expire after 15 minutes of inactivity and are lost when the server restarts.

⚡ User Login Page

User Login authenticates an existing Zendesk end-user into the messaging widget in one step — just email and password. It uses environment variables set on the hosting platform (or credentials saved on the Configuration page), so credentials survive restarts and redeploys securely.

How it works

1
You enter your email and password. The app sends them to POST /api/quick-login.
2
The server looks up the Zendesk user by email via GET /api/v2/users/search.json?query=email using admin API credentials from environment variables.
3
The server generates a JWT using your Sunco Key ID and Secret Key. The JWT contains scope: "user", external_id, name, email, and email_verified: true, signed with HMAC-SHA256.
4
The frontend stores the JWT, user info, and widget key in sessionStorage, then redirects to /?wk=WIDGET_KEY.
5
The server intercepts the request and injects the Zendesk snippet script into the HTML <head>. This is critical — loading snippet.js dynamically causes an InboundFilters crash.
6
On page load, the auto-login code detects the ?wk= parameter, retrieves auth data from sessionStorage, and calls zE('messenger', 'loginUser', jwtCallback, loginCallback) to authenticate.
7
On success, the messaging widget opens and the user can chat. On failure, an error message is displayed.
💡 First time? If environment variables are not configured, you'll see a warning on the User Login page. Either set ZENDESK_SUBDOMAIN, ZENDESK_ADMIN_EMAIL, ZENDESK_API_TOKEN, SUNCO_APP_ID, SUNCO_KEY_ID, SUNCO_SECRET, and ZENDESK_WIDGET_KEY on the hosting platform, or save credentials on the Configuration page.

📋 API Logs Page

The API Logs page displays every Zendesk/Sunco API call made by the server, including the HTTP method, endpoint, status code, and expandable request/response bodies. Logs are auto-refreshed after every API call via a fetch interceptor — you don't need to click Refresh manually.

Sensitive fields (password, apiToken, secret, authorization) are automatically redacted to •••••••• in the logs.

🏗️ How the Code Works

Architecture

The app consists of two files:

  • server.js — Express backend that proxies all Zendesk/Sunco API calls (avoiding CORS), signs JWTs, stores credentials server-side, and injects the Zendesk snippet into the HTML.
  • index.html — Single-page app with four pages (Configuration, User Login, API Logs, Docs). All state management happens client-side, but credentials are never stored in the browser.

JWT Generation

The JWT is generated server-side in the generateJWT() function using Node.js's built-in crypto module. No external JWT library is used — the signing is done manually with crypto.createHmac('sha256', secret):

Header:  { alg: "HS256", typ: "JWT", kid: KEY_ID }
Payload: { scope: "user", external_id: "...", name: "...", email: "...",
           email_verified: true, iat: NOW, exp: NOW + 3600 }
Sign:    HMAC-SHA256(secret_as_raw_string, header_b64url + "." + payload_b64url)

The kid (Key ID) in the header tells Zendesk which API key was used to sign the token. The scope: "user" claim is required for messenger authentication — using "end_user" will cause a login failure.

Widget Snippet Injection

The Zendesk messaging widget (snippet.js) cannot be loaded dynamically via document.createElement('script') after page load — this causes an InboundFilters crash in the SDK's module loader. Instead, when User Login succeeds, the page redirects to /?wk=WIDGET_KEY&sd=SUBDOMAIN. The server intercepts this request, reads index.html, injects the <script id="ze-snippet"> tag into <head>, and serves the modified HTML. This way the snippet is part of the initial HTML parse, exactly as Zendesk intends.

Fetch Interceptor

The frontend replaces window.fetch with an interceptor that schedules an API log refresh after every non-/api/logs call. This ensures the API Logs page stays current without manual refresh. The original fetch is preserved as _rawFetch for internal use (so log refreshes don't trigger recursive refreshes).

🔒 Security Notes

  • Credentials are stored in environment variables — never in config.json or client-side storage. Set ZENDESK_* and SUNCO_* env vars on your hosting platform.
  • JWT signing is server-side — the Secret Key is never sent to the browser. Only the signed JWT token is returned.
  • API logs require authentication — all /api/ endpoints require a valid session or API key header.
  • API logs redact sensitive fields — passwords, API tokens, secrets, and authorization headers are replaced with ••••••••.
  • Sessions expire after 15 minutes of inactivity — in-memory credential store has a TTL to limit exposure.
  • JWTs expire after 15 minutes — reduced from 1 hour. A jti (JWT ID) blocklist supports revocation.
  • Rate limiting — authentication and user-creation endpoints are rate-limited (5–20 req/15min per IP).
  • Security headers — helmet middleware sets CSP, X-Frame-Options, X-Content-Type-Options, and more.
  • Input validation — email, name, external ID, and password fields are validated and sanitized before use.
  • Production error handling — stack traces are never leaked to clients in production mode.
